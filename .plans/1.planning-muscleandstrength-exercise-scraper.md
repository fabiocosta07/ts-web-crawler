# Plan: Muscle & Strength exercise scraper

**Date:** 2026-04-18
**Status:** draft

## Context

We want a routine that crawls `https://www.muscleandstrength.com/exercises` and
extracts, for every exercise listed on the site:

- The exercise **name**
- The **category** it belongs to (muscle group / body part — e.g. Chest, Back,
  Shoulders, Legs, Biceps, Triceps, Abs, Forearms, Glutes, Traps, Calves, etc.)
- One or more **photo URLs** (thumbnail shown in the listing and/or the main
  image on the exercise detail page)

The goal is to seed the Easy Workout domain model with a clean catalog of
exercises-by-category that the main app can reference, rather than scraping it
on demand. The existing generic crawler in `src/crawler/engine.ts` is a good
reference for HTTP + cheerio patterns but is purpose-built for BFS link
discovery — it does not extract domain data, so we need a dedicated routine.

A quick probe via `WebFetch` against the index page returned **HTTP 403**,
which strongly suggests the site filters on `User-Agent` / other headers. This
will need to be handled before any scraping works at all (see Open Questions).

## Goals

- Produce a self-contained scraper routine that can be run on demand (CLI or a
  protected API endpoint) and writes the results to Postgres via Prisma.
- Persist exercises grouped by category, deduplicated, with source URLs kept
  for traceability and re-crawling.
- Keep the scraper polite: configurable delay between requests, sensible
  User-Agent, concurrency cap, and obey any robots.txt rules we choose to
  honour.
- Separate the scraper from the generic BFS crawler engine — they have
  different responsibilities and different persistence shapes.

## Non-goals

- No exercise instructions / descriptions / muscles-worked / difficulty in
  v1. We can add those later once the name+category+photo skeleton works.
- No video URLs in v1.
- No downloading of image bytes; we only store the remote URL. Hosting a local
  mirror is a follow-up if we find the site blocks hotlinking.
- No scheduling / recurring refresh — manual trigger only for v1.
- Not integrated with the existing `CrawlJob` / `CrawlResult` flow. This is a
  separate, structured-scraper concern.

## Approach

### 1. Data model (Prisma)

Add two new models alongside the existing crawl tables:

```prisma
model ExerciseCategory {
  id         String     @id @default(uuid())
  slug       String     @unique          // e.g. "chest", "back"
  name       String                      // e.g. "Chest"
  sourceUrl  String                      // category listing URL on m&s
  createdAt  DateTime   @default(now())
  updatedAt  DateTime   @updatedAt
  exercises  Exercise[]
}

model Exercise {
  id          String            @id @default(uuid())
  slug        String            @unique  // derived from detail URL
  name        String
  sourceUrl   String                     // detail page URL on m&s
  imageUrl    String?                    // primary photo
  thumbnailUrl String?                   // listing thumbnail if different
  categoryId  String
  category    ExerciseCategory  @relation(fields: [categoryId], references: [id], onDelete: Cascade)
  scrapedAt   DateTime          @default(now())
  updatedAt   DateTime          @updatedAt

  @@unique([categoryId, slug])
}
```

A new migration `add-exercise-catalog` creates these tables without touching
the existing crawl tables.

### 2. Scraper module layout

```
src/scrapers/muscleandstrength/
  index.ts                 // runMuscleAndStrengthScrape() — public entry
  http.ts                  // fetchHtml() with UA, retry, delay, timeout
  categories.ts            // discoverCategories(): scrape the index page
  exercises.ts             // scrapeCategory(): paginated listing -> items
  detail.ts                // optional: scrapeDetail() for canonical image
  persist.ts               // upsert helpers (category + exercise)
  types.ts                 // DiscoveredCategory, DiscoveredExercise, ...
```

Keeping this under `src/scrapers/` (not `src/crawler/`) signals that it is a
**structured scraper**, not the generic BFS engine.

### 3. Crawl flow

1. **Fetch the index page** `https://www.muscleandstrength.com/exercises`.
2. **Discover categories** by selecting the "browse by muscle group" /
   body-part navigation. For each entry capture `{ name, slug, sourceUrl }`
   and upsert into `ExerciseCategory`.
3. **For each category**, fetch the listing page(s):
   - Parse each exercise card to get `{ name, detailUrl, thumbnailUrl }`.
   - Follow pagination (`?page=2`, `?page=3`, …) until an empty page or a
     404 is reached.
4. **Optionally fetch each detail page** to get the canonical / higher-res
   `imageUrl`. Gate this behind a `fetchDetails: boolean` option because it
   multiplies request volume by ~N exercises.
5. **Upsert** each exercise using `(categoryId, slug)` as the conflict key so
   re-runs are idempotent.
6. **Report** counts per category and total at the end (structured log).

### 4. Politeness & robustness

- Custom `User-Agent`: `"easy-workout-scraper/0.1 (+contact@example.com)"` —
  the default `ts-web-crawler/1.0` UA likely causes the 403 we observed.
- Accept header: `text/html,application/xhtml+xml`.
- Request delay: configurable, default **1000 ms** between requests to the
  same host.
- Concurrency: **1** (serial) for v1. Parallelism is a follow-up once we
  confirm the site tolerates it.
- Retry: exponential backoff for 429 / 5xx up to 3 tries; abort on persistent
  4xx (other than 429).
- Timeout: reuse the 10 s pattern from `engine.ts`.
- Respect `robots.txt` — fetch it once and short-circuit disallowed paths.

### 5. Invocation

Two options (non-exclusive):

- **CLI script:** `npm run scrape:ms` → `tsx src/scrapers/muscleandstrength/cli.ts`.
  Simplest for one-off catalog seeding.
- **Protected API endpoint:** `POST /api/scrape/muscle-and-strength` guarded
  by the existing `authMiddleware`, kicks off the scrape in the background
  like the existing `/crawl-jobs` POST. Returns a job ID; we'd add a
  `ScrapeJob` model or reuse `CrawlJob` with a `kind` discriminator.

**Recommendation:** start with the CLI only. Wire the endpoint later if we
actually need on-demand triggering from the UI.

### 6. Testing

- Unit tests for the cheerio parsers — feed in saved HTML fixtures captured
  from the real site (check a few category pages + one detail page into
  `src/tests/fixtures/muscleandstrength/`). No network in unit tests.
- Integration test for `persist.ts` against the Testcontainers Postgres
  already wired up, asserting idempotency of re-runs.
- Skip a full live e2e — rate-limited by the site and flaky by nature. A
  smoke script that hits one category with `--dry-run` (no DB writes) is
  enough.

### 7. Alternatives considered

- **Extend the generic `runCrawl` engine with selectors.** Rejected: mixes
  BFS link discovery with structured extraction and would pollute
  `CrawlResult` with fields it doesn't need.
- **Use a headless browser (Playwright).** Rejected for v1 — the listing
  pages appear to be server-rendered HTML, and cheerio + fetch is already in
  the stack. Revisit only if we find critical data is client-rendered.
- **Scrape a public exercise dataset / API instead.** Worth checking (e.g.
  wger, ExerciseDB). If a permissively-licensed dataset exists, it is almost
  certainly a better primary source than scraping. Flagged as an open
  question below.

## Open questions

- Is there a licensing / ToS concern with scraping muscleandstrength.com at
  scale? We should read their `robots.txt` and ToS before running the full
  scrape.
- Is a public dataset (wger, Free Exercise DB on GitHub, ExerciseDB) a better
  primary source? If yes, this plan may be replaced by an import routine.
- What exactly does the listing page DOM look like? We hit 403 on first fetch
  — we need to confirm the selectors by capturing real HTML from a browser
  once scraping is unblocked (UA change).
- Do we need localized names, or English-only? Assume English-only for v1.
- Should we store multiple images per exercise (start / mid / end position)?
  Probably yes eventually — the schema above can grow a separate
  `ExerciseImage` table without a breaking migration.
